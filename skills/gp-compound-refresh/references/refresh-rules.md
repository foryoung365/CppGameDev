# Refresh Rules

Use these rules when maintaining the host project's gameplay experience library.

## Action Definitions

### Keep

Use when the doc still matches verified code and verified workflow reality.

### Update

Use when the core learning remains correct but one or more of these drifted:

- module names
- config names
- file or class references
- terminology
- related links

Only apply the update when the new values are explicit and evidenced.

### Consolidate

Use when two or more docs say the same verified thing and one canonical doc can hold the unique value safely.

Merge only the unique verified material, then delete the redundant doc.

### Delete

Use when the learning no longer reflects real verified code or workflow reality and no trustworthy narrower update or consolidation makes sense.

Delete directly. Do not create archive folders.

## Forbidden Moves

- inferred successor docs
- speculative rewrites
- replacement-by-guess
- archiving instead of deciding
- converting uncertainty into guidance

## Evidence Standard

If the current state is not fully evidenced, do not change the doc.

This is especially strict for:

- root cause changes
- guidance reversals
- deletions
- workflow guardrail reversals

When in doubt, preserve the existing doc and report that the refresh could not be completed safely.
